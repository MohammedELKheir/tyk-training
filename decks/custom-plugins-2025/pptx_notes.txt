"By leveraging custom plugins, you can meet unique business requirements such as:
Custom rate limiting logic.
Integration with external services.
Real-time request and response transformation."

SLIDE 4 - FULL NOTES

Overview of the Table:
"Here, I’ve summarized the key middleware hooks, their purpose, and common use cases

Hook Types and Execution Phases
Pre (Request) Hook – Request Plugin
"The Pre hook runs before the reverse proxy processes the request. This is the first middleware to execute."
"It’s commonly used for tasks like IP rate limiting or request enrichment."
Example Use Case:
"You can use this hook to validate headers or inject metadata into the request before passing it further."

Authentication Hook – Auth Plugin
"This hook runs before the reverse proxy as well, but it specifically handles authentication."
"It replaces the default Tyk authentication flow, allowing you to integrate custom authentication logic, such as legacy auth systems."
Example Use Case:
"You can validate tokens against external identity providers or custom databases."

Post-Auth (Request) Hook – Auth Plugin
"The Post-Auth hook runs after the authentication middleware but still before the reverse proxy."
"It allows you to add special custom authentication logic."
Example Use Case:
"You can use it to implement additional authorization checks, like verifying user roles or permissions."

Post (Request) Hook – Request Plugin
"The Post hook is executed at the final request phase before the reverse proxy sends the request to the upstream API."
"It’s ideal for modifying the request, such as adding headers or transforming payloads."
Example Use Case:
"You can dynamically append headers, like an API key or a custom tracking ID."

Response Hook – Response Plugin
"The Response hook runs after the upstream API sends the response back to Tyk."
"It allows you to modify the response before it reaches the client."
Example Use Case:
"You can obfuscate sensitive data, format JSON responses, or inject additional headers like caching policies."

Analytics Hook – Analytics Plugin
"The Analytics hook operates at the final phase of request or response processing."
"It’s specifically used for logging and analytics."
Example Use Case:
"You can obfuscate sensitive data in logs, ensuring compliance with data protection regulations."

Key Takeaways:
"By using these hooks, you gain full control over the request and response lifecycle."
"This is particularly useful for scenarios like custom authentication, rate limiting, or response transformation."
"Each hook has its own specific execution phase, allowing you to optimize your middleware logic effectively."

"Python plugins are a great choice for teams already familiar with Python and looking to quickly extend Tyk’s behavior."

"You can implement logic for advanced logging, API transformations, or even dynamic authentication."

Key Takeaways:
"Each plugin type has its advantages depending on your use case and development preferences."

"Let’s take a look at how a typical request flows through the Tyk Gateway and how plugins come into play to process the request."
Step-by-Step Explanation:
Step 1: The Client Sends a Request
"The process starts with a client sending an HTTP request to an API managed by the Tyk Gateway."
"This could be any type of client—an application, a web browser, or even a mobile app."
Step 2: Tyk Gateway Processes the Request
"The Tyk Gateway acts as the entry point for all incoming requests."
"It first evaluates the API request and checks for any configurations applied to that API, like authentication, rate limiting, or access policies."
Step 3: Plugins Execution
"Once the request reaches the gateway, any configured custom plugins are executed."
"Plugins can perform tasks such as:"
Custom Authentication: Validate tokens or credentials.
Data Transformation: Modify request payloads or headers.
Rate Limiting Enforcement: Check if a request exceeds the rate limits."
"Plugins are essential for extending Tyk's capabilities with custom logic tailored to your needs."
Step 4: Forwarding to Upstream Services
"Once the plugins have processed the request, Tyk forwards the modified request to the upstream service or backend API."
"At this point, the request is fully ready and contains any transformations or additional checks performed by the plugins."
Step 5: Response Back to the Client
"The upstream service processes the request and returns a response to the Tyk Gateway."
"Tyk Gateway forwards the upstream response back to the client, completing the request-response cycle."


Key Takeaway:
"Tyk acts as a smart intermediary, not just forwarding requests but also performing critical operations using plugins."
"This modular approach ensures that you can handle complex use cases like authentication, rate limiting, and data manipulation if needed

Introduction:
"Tyk provides flexibility with a few  deployment options: Local Plugins, Plugin Bundles, and gRPC Plugins."
1. Local Plugins
"Local Plugins are the simplest option."
"Here, the plugin source code and its configuration are stored directly on the same file system as the Tyk Gateway."
"Configuration is defined within the API Definition, and the plugins execute locally."
"This is ideal for environments where the gateway and plugins reside on the same machine."

2. Plugin Bundles (Remote)
"In the second option, Plugin Bundles allow you to bundle your plugin source code and configuration into a zip file."
"These bundles are then uploaded to a remote web server."
"The Tyk Gateway will download, extract, cache, and execute these plugins as part of the API lifecycle."
"This approach works well for distributed systems or when plugins are frequently updated."
Example Use Case:
"For production setups where plugins are managed centrally and shared across multiple Tyk Gateways."

3. gRPC Plugins (Remote)
"Lastly, Tyk supports gRPC-based remote plugins."
"With gRPC Plugins, your custom plugins are hosted on a remote server and executed via gRPC."
"The Tyk Gateway sends requests to the gRPC server during specific points in the API request/response lifecycle."
"This allows you to use any programming language to develop your plugins, not just Go or JavaScript."
Example Use Case:
"For enterprises that need high performance, remote plugin execution, or use other languages
Key Takeaways:
"To summarize, Local Plugins are simple and fast but tied to the Gateway's file system."
"Plugin Bundles offer central management for shared plugins across multiple gateways."
"gRPC Plugins provide the most flexibility with remote execution and multi-language support."

What are Local Plugins?
"Local Plugins are custom plugins where the source code is stored and executed directly on the Tyk Gateway."
"This approach ensures low latency and minimal dependencies."

Pros of Local Plugins:
Performance
"Because the plugin code is executed locally, the performance is high with minimal latency."
"This is ideal for real-time operations where speed matters."
Simplicity
"Local plugins are easy to configure and manage since everything resides within the Tyk Gateway."
No Network Dependencies
"You don’t need to worry about external communication or connectivity issues, making local plugins a good fit for isolated environments like on-prem setups."

Cons of Local Plugins:
Limited Scalability
"Local plugins don’t scale well in distributed environments or across multiple Tyk Gateway instances."
"If you need to reuse plugins across a fleet of gateways you will have to compile for each gateway that you are using
Harder to Manage Updates
"Every time you make an update to the plugin, you need to manually update it on each gateway."
"This can become time-consuming, especially as the infrastructure grows."

Summary:
"To summarize, Local Plugins are great for simplicity and performance, especially isolated environments."

"gRPC Plugins allow you to write and run plugins on a remote server using the gRPC protocol."

Pros of gRPC Plugins:
Language Flexibility

Scalability

How Plugin Bundles Work
"A plugin bundle is a ZIP file that contains the source code and the manifest.json file."
"The API definition references the plugin bundle name to identify it."
"The Tyk Gateway takes care of downloading, caching, extracting, and executing the plugin from the bundle."
Bundle Caching
"For performance, Tyk caches plugin bundles locally."
"To ensure updates to a plugin bundle, you can use unique filenames, such as version numbers."
"The cached location is: {TYK_ROOT}/{CONFIG_MIDDLEWARE_PATH}/bundles."
Gateway Configuration
"To enable plugin bundles, you need to update the tyk.conf file with the following configurations:

Explanation:
enable_bundle_downloader: Enables the Tyk Gateway to download bundles from a specified URL.
bundle_base_url: The base URL where the plugin bundles are stored.
public_key_path: Path to the public key for verifying signed bundles.
enable_coprocess: Enables advanced plugin execution features, such as rich plugins.

Introduction"Next, we’ll discuss the manifest files within plugin bundles and their importance in configuring plugins for Tyk."
What is a Manifest File?
"A manifest file is a critical configuration file for your plugin bundle."
"It defines which source code files are included in the bundle
"If a file isn’t listed in the manifest, it will be ignored, even if it’s present in the plugin directory."
Key Components of the Manifest File
file_list Block
"This block specifies the source files to include in the plugin bundle. 
custom_middleware Block
"This block allows you to define pre and post execution middleware.For instance:"custom_middleware": {
    "pre": [ { "name": "PreMiddleware" } ],
    "post": [ { "name": "PostMiddleware" } ]
}

The pre block runs before the request is proxied to the upstream.
The post block runs after receiving a response from the upstream."
driver Field
"The driver specifies the programming language used for the plugin.
checksum and signature Fields are usually autofilled during the build process
"These fields ensure the integrity of the plugin bundle:
checksum: Automatically generated during the build process to validate the content.
signature: Provides an additional layer of security to confirm the bundle is trusted."

Why is the Manifest File Important?
"The manifest file ensures that:
Only the required source files are included in the bundle.
Execution behavior (pre/post middleware) is explicitly defined.
The plugin bundle is secure and tamper-proof through checksums and signatures."
